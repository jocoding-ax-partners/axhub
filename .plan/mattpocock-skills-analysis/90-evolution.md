## 분석 가치 (왜 evolution 이 중요)

### 1. Deprecated 의 후계 매핑이 명확해짐
원본 `05-excluded-skills.md` 는 추론으로 후계 매핑했어요 — 이제 git history 가 confirms:

| Deprecated | 분석 doc 의 추론 | Git history 확인 |
|---|---|---|
| `design-an-interface` | → `improve-codebase-architecture/INTERFACE-DESIGN` | ✓ (직접 trace 못 봄, 그러나 INTERFACE-DESIGN.md 가 후속 commit 에 추가) |
| `qa` | → `triage` | ✓ (`62f43a1` 에서 deprecated 로 이동) |
| `request-refactor-plan` | → `to-prd + to-issues` | ✓ (`62f43a1` 에서 deprecated 로 이동, `to-prd` + `to-issues` 동시 추가) |
| `ubiquitous-language` | → `grill-with-docs/CONTEXT-FORMAT` | ✓ (`fb847c6` `domain-model → grill-with-docs` 와 함께 deprecated) |

추가 사라진 것:
- `prd-to-plan` (분석 doc 에서 누락) — `a77fa6e` 에서 명시적 제거
- `triage-issue` (분석 doc 에서 누락) — `a32ebfb` 에서 제거, `triage` 가 흡수

### 2. 이름 변경 패턴이 매트의 사고 변화 보여줌
- `write-a-prd` → `to-prd`: imperative ("써라") → 결과 표현 ("PRD 로")
- `github-triage` → `triage`: tracker-specific → tracker-agnostic
- `domain-model` → `grill-with-docs`: noun-based → action-based

세 변경이 같은 방향:
> "무엇을 만드는가" 보다 "어떻게 사용하는가" 가 명확.

### 3. Bucket migration 시점이 정책 명확화 시점
`62f43a1` 에서 deprecated bucket 도입 → `51384f4` 에서 plugin.json 에서 deprecated 제거.

이 순서 = "버리지는 않고 분리만" 의 의도 명시. 흡수된 skill 도 archive 로 보존.

### 4. setup-matt-pocock-skills 가 마지막에 도입
hard/soft dependency 분리 (ADR-0001) 도 setup 도입 후. 즉 매트가:
- 먼저 7 consumer skill 만듦
- 사용해보면서 setup 패턴 필요 발견
- setup-matt-pocock-skills 로 통합
- ADR-0001 로 dependency 분리 명시

→ skill 자체가 emerge 했다. 미리 디자인 X.

### 5. `.out-of-scope/question-limits.md` 가 HEAD commit
가장 최신 변경이 question-limits 메모. 매트가 issue #44 ("Codex just asked me 200 questions") 의 "고정 cap 추가" 요청을 거절했고, 그 거절 이유 자체를 자기 패턴 (`.out-of-scope/`) 으로 capture.

self-application 의 살아있는 예 — 자기 거절 결정도 자기 매커니즘으로 보존.
