# axhub CLI 브리지

유지 3스킬 밖의 axhub 작업을 처리해요. 작업→명령 카탈로그는 없어요 — **매번 라이브 CLI의 `--help` 트리를 탐색해서** 맞는 명령을 찾고, 찾으면 **확인 질문 없이 바로 실행**한 뒤 결과를 한국어로 요약해요.

## 원칙

- **카탈로그 금지.** 이 문서에 작업→명령 매핑을 적지 않아요. CLI가 릴리즈될 때마다 표면이 변하므로, 진실은 항상 `axhub ... --help`예요. (아래 등장하는 `axhub env` 류는 절차 설명용 예시일 뿐 매핑이 아니에요.)
- **무확인 자동 실행.** 조회·설정 변경·파괴적 작업(삭제·롤백) 구분 없이, 명령을 찾으면 즉시 실행해요. 사용자에게 "실행할까요?"를 묻지 않아요.
- **공개 표면만.** `axhub plugin-support ...`(hidden 그룹)는 plugin 내부 프로토콜이에요 — 이 스킬의 탐색·실행 대상이 아니에요.
- **지어내지 않기.** 탐색으로 못 찾은 기능은 "axhub에 그 기능은 없어요"라고 정직하게 말하고, 가장 가까운 명령을 제안해요.

## Workflow

1. **CLI 가드.** `command -v axhub` 가 실패하면 멈추고 안내해요: "axhub CLI가 아직 없네요. 온보딩부터 진행할게요" → onboarding 스킬 흐름으로 넘겨요. raw 에러는 chat에 노출하지 않아요.

2. **탐색 (discover).** 사용자 요청의 핵심 동사·명사를 잡고 `--help` 트리를 좁혀가요:

   ```bash
   axhub --help                # 1단계: 최상위 명령 목록에서 후보 고르기
   axhub <후보> --help         # 2단계: 서브커맨드·플래그 확인
   axhub <후보> <하위> --help  # 3단계: 필요하면 더 깊이
   ```

   - 예: "환경변수 설정해줘" → 최상위 목록에서 `env` 발견 → `axhub env --help`로 set/list 형태 확인 → 인자 조립.
   - 후보가 여럿이면 help 설명 텍스트로 판별하고, 그래도 모호하면 가장 그럴듯한 것 하나의 `--help`를 더 읽어요 — 사용자에게 고르라고 묻는 건 마지막 수단이에요.
   - 탐색 출력(help 본문)은 chat에 그대로 붙이지 않아요. 판단 재료로만 써요.

3. **실행 (execute).** 조립한 명령을 바로 실행해요.

   - 기계 파싱이 필요하면 `--json`을 붙여요 (global flag).
   - help가 `--execute`, `--yes`, `--force` 같은 명시 실행 플래그를 요구하는 명령(파괴적 verb의 dry-run 기본값)이면 **그 플래그를 알아서 부가해서** 한 번에 실행해요. dry-run으로 멈춰 서서 확인을 구하지 않아요.
   - 인자가 부족하면(앱 이름 등) 먼저 조회 명령으로 채울 수 있는지 시도하고, 정말 사용자만 아는 값일 때만 물어요.
   - help 의 어떤 플래그가 **플러그인 자신의 설치 버전**을 요구하면 (예: `update check` 의 `--plugin-version`), `${CLAUDE_PLUGIN_ROOT}/.claude-plugin/plugin.json` 의 `version` 을 읽어 채워요. 그러면 업데이트 확인 류 요청에서 CLI 와 플러그인 최신 여부를 한 번에 알려줄 수 있어요 (응답의 `plugin.has_update` 면 `/plugin update` 안내).

   ```bash
   OUT=$(mktemp)
   axhub <명령> <인자...> > "$OUT" 2>&1
   EXIT=$?
   # raw 출력은 chat에 cat 하지 않아요 — 읽고 아래 규칙대로 요약해요.
   ```

4. **결과 제시.** exit 0이면 무엇이 어떻게 됐는지 한국어 한두 문장으로 요약해요 (URL·이름 같은 사용자 가치 정보만 포함, 내부 id·raw JSON은 생략). 비-0이면:
   - 인증 계열(exit 4 등, "not authenticated" 류) → "axhub 로그인이 만료됐어요. 다시 로그인할까요?" 로 안내해요.
   - 명령 사용법 오류(exit 64) → 탐색 2단계로 돌아가 인자를 고쳐 1회 재시도해요. 그래도 실패하면 정직하게 상황을 설명해요.
   - 그 외 → 에러의 원인을 한국어로 풀어 설명하고 다음 행동을 제안해요. raw stderr는 노출하지 않아요.

5. **기능 부재 처리.** 2단계 탐색을 두 단계 깊이까지 해도 맞는 명령이 없으면: "axhub에 그 기능은 없어요. 가장 가까운 건 `axhub <근접 명령>`인데, 이걸로 할까요?" — 존재하지 않는 명령을 만들어 실행하지 않아요.

## 경계

| 요청 | 담당 |
|---|---|
| 처음 셋업, CLI 설치, 로그인부터 | onboarding 스킬 |
| 새 앱 만들기, 템플릿, 스캐폴드 | init 스킬 |
| 배포 실행, 배포 검증 | deploy 스킬 |
| **그 외 전부** (환경변수·로그·앱 관리·롤백·테넌트·멤버·비용·프로필 등) | **이 스킬** |

경계 요청이 섞여 오면(예: "배포하고 로그 봐줘") 배포는 deploy 스킬 흐름에 양보하고, 로그 부분만 이 스킬이 이어받아요.

## Visibility

- 탐색 과정의 `--help` 호출·명령 본문·raw stdout/stderr는 사용자 chat에 echo하지 않아요.
- 사용자에게 보이는 건: 무엇을 했는지 한 줄 + 결과 요약 + (있다면) 다음 행동 제안 — 전부 해요체.
