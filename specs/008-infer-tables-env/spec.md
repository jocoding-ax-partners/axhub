## User Scenarios & Testing *(mandatory)*

### User Story 1 - 개발한 코드에서 필요한 테이블·환경변수 추천 받기 (Priority: P1)

개발자가 앱 코드를 작성한 뒤, 시스템이 그 소스코드를 분석해서 이 앱이 필요로 하는 것으로 보이는 (a) 데이터 테이블 목록(테이블 이름 + 후보 컬럼)과 (b) 환경변수 목록을 만들어 보여줘요. 각 항목에는 "코드의 어디에서 그렇게 추론했는지" 근거가 함께 붙어서 개발자가 검증할 수 있어요. 이 단계에서는 아무것도 바꾸지 않고 읽기 전용으로 추천만 해요.

**Why this priority**: 이 기능의 핵심 가치예요. "앱은 다 만들었는데 이제 어떤 데이터 구조랑 설정을 잡아야 하지?"를 자동 체크리스트로 바꿔줘요. 이 한 가지(추천만)만 있어도 독립적으로 가치를 주는 MVP 예요.

**Independent Test**: 데이터 모델과 설정 읽기가 들어있는 프로젝트를 대상으로 분석을 실행해서, 기대하던 테이블·컬럼과 환경변수가 근거와 함께 나열되는지, 그리고 아무것도 변경되지 않는지 확인하면 돼요.

**Acceptance Scenarios**:

1. **Given** 데이터 엔티티/모델이 정의된 프로젝트, **When** 사용자가 분석을 요청하면, **Then** 시스템은 추천 테이블 목록을 각 테이블의 추론 컬럼과 "추론 근거가 된 코드 위치"와 함께 제시한다.
2. **Given** 설정값/시크릿을 읽는 코드가 있는 프로젝트, **When** 사용자가 분석을 요청하면, **Then** 시스템은 추천 환경변수 목록을 근거 위치 및 "시크릿으로 보이는지" 여부와 함께 제시한다.
3. **Given** 감지 가능한 데이터 모델이나 설정 사용이 전혀 없는 프로젝트, **When** 분석이 실행되면, **Then** 시스템은 추론된 것이 없음을 분명히 말하고 다음에 할 수 있는 일을 제안한다.

### User Story 2 - 승인하면 추천을 바로 적용 (Priority: P2)

개발자가 추천을 검토한 뒤 승인하면, 시스템은 무엇이 생성·변경될지 정확히 보여주는 최종 확인을 거친 후, 기존의 테이블 관리·환경변수 관리 기능으로 위임해서 추천된 테이블을 만들고 환경변수 키를 등록해줘요. 테이블은 스키마를 이미 알고 있으니 바로 만들고, 환경변수는 값을 추론할 수 없으므로 키만 잡아두고 각 값은 사용자가 안전한 입력(stdin)으로 직접 넣게 안내해요.

**Why this priority**: 추천에서 끝나지 않고 "실제 동작하는 설정"까지 한 번에 닫아줘요. P1 의 추천 결과에 의존하지만, 적용 자체는 독립적으로 테스트할 수 있어요(알려진 추천을 승인 → 리소스 생성 확인).

**Independent Test**: 확정된 추천을 승인해서 테이블이 존재하고 환경변수가 설정됐는지 확인하고, 거절했을 때는 아무것도 바뀌지 않는지 확인하면 돼요.

**Acceptance Scenarios**:

1. **Given** 추천이 표시된 상태, **When** 사용자가 승인하면, **Then** 시스템은 적용 직전에 최종 미리보기(대상 앱, 정확한 테이블/컬럼, 환경변수 키 — 시크릿 값은 마스킹)를 보여주고 명시적 확인을 다시 요구한다.
2. **Given** 사용자가 확인함, **When** 적용이 실행되면, **Then** 추천된 각 테이블은 생성되고 각 환경변수는 키가 등록되며(값은 사용자가 stdin 으로 입력하거나 건너뜀), 항목별 성공/실패/건너뜀 요약이 반환된다.
3. **Given** 사용자가 확인 게이트에서 거절함, **When** 적용이 중단되면, **Then** 어떤 테이블이나 환경변수도 생성·변경되지 않는다.
4. **Given** 시크릿을 담는 환경변수, **When** 적용할 때, **Then** 그 값은 미리보기나 로그 어디에서도 평문으로 표시되지 않는다.

### User Story 3 - 개발 흐름 중 자동 제안 (Priority: P3)

개발 라이프사이클의 자연스러운 지점(예: 새 앱을 막 스캐폴드한 직후, 또는 배포 직전)에서, 시스템이 명시적 요청을 기다리지 않고 먼저 "코드를 분석해서 필요한 테이블·환경변수를 추천해드릴까요?" 하고 비차단(non-blocking) 제안을 해줘요. 이 자동 제안 시점에는 무거운 전체 소스 분석을 돌리지 않고 경량 신호만 보는 넛지로 그치고, 전체 분석은 사용자가 그 제안을 수락한 뒤에만 실행해요.

**Why this priority**: 편의성·발견성 레이어예요. 이 기능이 가장 유용한 순간에 스스로 모습을 드러내요. P1/P2 위에 얹는 선택적 층이라 우선순위가 가장 낮아요.

**Independent Test**: 해당 라이프사이클 순간을 발생시켜서, 분석을 제안하는 비차단 안내가 뜨는지, 그리고 그 제안을 거절해도 부작용이 전혀 없는지 확인하면 돼요.

**Acceptance Scenarios**:

1. **Given** 사용자가 관련 라이프사이클 순간에 도달함, **When** 그 순간이 발생하면, **Then** 시스템은 강제하지 않고 테이블/env 분석을 제안한다.
2. **Given** 사용자가 자동 제안을 무시하거나 거절함, **When** 계속 진행하면, **Then** 사용자의 작업 흐름은 그대로 바뀜 없이 이어진다.
3. **Given** 자동 제안 시점, **When** 넛지를 표시하면, **Then** 시스템은 전체 소스 분석을 돌리지 않고 경량 신호만 확인하며, 전체 분석은 사용자가 수락한 뒤에만 시작한다.

---

### Edge Cases

- 코드의 모델이 외부 연결 데이터 소스(앱 소유 동적 테이블이 아닌)를 가리킬 때 — 앱 소유 데이터 모델과 외부 커넥터 읽기를 구분해서, 외부 소스는 새 테이블로 추천하지 않는다.
- 컬럼 타입이나 제약(필수/고유/PK)을 확신할 수 없을 때 — 베스트게스로 추천하되 "검토 필요"로 표시한다.
- 환경변수가 이미 설정돼 있거나 테이블이 이미 존재할 때 — 추천에서 "이미 설정됨"으로 표시하고, 적용 시 건너뛴다(멱등).
- 소스에 시크릿 값이 하드코딩돼 있을 때 — 보안 발견("환경변수로 옮기세요")으로 플래그하고, 그 시크릿 평문을 표시되는 추천에 절대 복사하지 않는다.
- env 키에 소스상 비시크릿 기본값이 있을 때(예: `getenv("PORT", "8000")`) — 그 기본값을 미리 채울 값으로 제안한다.
- 코드로만 정의된 모델(선언적 스키마 없이 흩어진 ORM 클래스·런타임 getenv) — 베스트에포트로 "검토 필요"로 표시하고 recall 보장 대상에서 제외한다.
- 매우 큰 코드베이스 / 모델이 아주 많을 때 — 스캔한 범위와 안 한 범위를 보고한다(조용한 truncation 금지).
- 신호가 충돌할 때(예: 두 파일이 같은 엔티티를 다르게 정의) — 조용히 하나를 고르지 않고 충돌을 드러낸다.
- 로그인 안 됨 / 대상 앱 미선택 — 읽기 전용 추천은 그래도 만들 수 있지만, 적용에는 인증과 대상 앱이 필요하다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 프로젝트 소스코드를 분석해서 앱이 필요로 하는 것으로 보이는 데이터 테이블 집합을 추론한다(각 테이블의 이름, 후보 컬럼, 그리고 컬럼별 타입과 기본 제약(필수/고유/PK 추정) 포함). axhub 템플릿 스택(Next.js/TypeScript 프런트, FastAPI/Python 백엔드)을 1급으로 지원하고, 그 외 스택은 베스트에포트로 처리한다.
- **FR-002**: System MUST 소스코드로부터 앱이 필요로 하는 것으로 보이는 환경변수 집합(설정값과 시크릿)을 추론한다.
- **FR-003**: 추론된 각 테이블·환경변수에 대해, System MUST 그 추론의 근거가 된 소스 위치와 패턴을 함께 제시해서 사용자가 검증할 수 있게 한다.
- **FR-004**: System MUST 추론된 각 환경변수를 시크릿/비시크릿으로 분류하고, 시크릿 값을 평문으로 표시하지 않는다.
- **FR-005**: System MUST 추천을 사람이 읽을 수 있는 요약으로 제시하되, (a) 새로 추천하는 항목, (b) 이미 설정된 항목, (c) 검토가 필요한 저신뢰 추측을 구분한다.
- **FR-006**: System MUST 어떤 테이블이나 환경변수를 생성·변경하기 전에 사용자의 명시적 승인을 요구한다. 승인이 없으면 동작은 읽기 전용이다.
- **FR-007**: 적용 직전에 System MUST 정확히 무엇이 생성·변경될지 최종 미리보기(대상 앱, 테이블 이름/컬럼, 마스킹된 환경변수 키)를 보여주고 명시적 확인을 받는다.
- **FR-008**: 승인 시 System MUST 기존의 테이블 관리·환경변수 관리 기능에 위임해서 추천을 적용하고, 항목별 성공/실패/건너뜀을 보고한다. 테이블은 추론된 스키마로 생성하고, 환경변수는 값을 추론하지 않으므로 키만 등록하되 각 값은 사용자가 안전한 입력(stdin)으로 제공하거나 건너뛸 수 있다.
- **FR-009**: 적용은 멱등이어야 한다 — 이미 존재하는 항목은 중복 생성하거나 안내 없이 덮어쓰지 않고 건너뛴다.
- **FR-010**: System MUST 관련 개발 라이프사이클 순간에 분석을 차단·강제하지 않고 먼저 제안하며, 거절은 어떤 부작용도 없어야 한다. 이 자동 제안 단계는 경량 비차단 넛지로만 동작하고, 무거운 전체 소스 분석은 사용자가 수락한 뒤에만 실행한다.
- **FR-011**: 아무것도 추론할 수 없을 때, System MUST 그 사실을 분명히 말하고 사용자가 다음에 할 수 있는 일을 제안한다.
- **FR-012**: System MUST 소스에서 발견한 하드코딩된 시크릿 리터럴을 표시되는 어떤 출력에도 복사하지 않고, 대신 발견으로 플래그하고 환경변수로 옮기길 권한다.
- **FR-013**: 커버리지가 제한될 때(예: 큰 코드베이스, 미스캔 영역), System MUST 전체를 분석한 것처럼 암시하지 않고 분석하지 못한 범위를 공개한다.
- **FR-014**: 변경 적용에는 사용자 인증과 대상 앱 선택이 필요하다. 둘 중 빠진 게 있으면 System MUST 적용 전에 그걸 해결하도록 안내한다(추천 자체는 읽기 전용으로 계속 제공).
- **FR-015**: System MUST 추천 결과를 휘발성으로 다룬다 — 추천은 표시만 하고 어떤 파일에도 영구 저장하지 않으며, 영구화되는 것은 사용자가 승인해 적용한 테이블·환경변수 변경뿐이다. 다시 추천이 필요하면 소스를 재분석한다.
- **FR-016**: System MUST 환경변수의 *값* 을 추론하거나 만들어내지 않는다. 소스로부터는 필요한 *키* 와 시크릿 여부만 도출하고, 값은 적용 단계에서 사용자 입력으로만 받는다. 단, 소스에 비시크릿 기본값이 드러난 경우(예: 기본 포트)는 미리 채울 값으로 제안할 수 있고, 하드코딩된 시크릿을 발견하면 그 키를 환경변수로 옮기도록 제안하되 값 자체는 표시하지 않는다.

### Key Entities *(include if feature involves data)*

- **Recommendation(추천)**: 한 프로젝트에 대한 전체 분석 결과 — Table Suggestion 집합 + Env Var Suggestion 집합 + 커버리지 메모를 담는다. 휘발성이며 파일로 영구 저장하지 않는다(세션 표시용).
- **Table Suggestion(테이블 추천)**: 추론된 동적 테이블 하나 — 이름, 후보 컬럼(이름 + 베스트게스 타입 + 기본 제약(필수/고유/PK 추정) + 신뢰도), 소스 근거, 상태(신규 / 이미 존재 / 검토 필요).
- **Env Var Suggestion(환경변수 추천)**: 추론된 환경변수 하나 — 키, 시크릿/비시크릿 분류, 소스 근거, 상태(신규 / 이미 설정 / 검토 필요). 시크릿 평문 값은 절대 담지 않는다.
- **Source Evidence(소스 근거)**: 어떤 추론을 정당화한 코드의 위치와 패턴.
- **Apply Plan(적용 계획)**: 사용자가 승인한 추천 부분집합 — 수행할 정확한 생성/설정 동작과 그 결과.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 선언적 스키마·환경 아티팩트(예: schema.prisma, 마이그레이션 파일, .env.example)를 사용하는 대표 fixture 앱 셋(axhub 템플릿 스택 Next.js/TypeScript + FastAPI/Python)에 대해, 개발자가 수동으로 설정했을 데이터 테이블·환경변수의 90% 이상이 추천에 잡힌다(높은 recall). 코드로만 정의된 모델(흩어진 ORM 클래스·런타임 getenv)은 베스트에포트로 처리하며 이 recall 보장 대상이 아니다.
- **SC-002**: 대표 fixture 셋 기준, 추천된 항목 10개 중 사용자가 거절하는 오탐(false positive)이 1개 이하다(precision).
- **SC-003**: "코드 작성 완료" 상태에서 "테이블 + 환경변수 설정 완료"까지, 추천과 한 번의 승인 적용으로 5분 이내에 도달한다(수동으로 하나씩 설정하는 것 대비).
- **SC-004**: 추천된 모든 항목이 검증 가능한 소스 근거를 보여준다 — 항목의 100%가 코드 위치로 추적 가능하다.
- **SC-005**: 어떤 추천·미리보기·로그에서도 시크릿 값이 평문으로 노출되지 않는다(0건).
- **SC-006**: 사용자의 명시적 확인 없이 생성·변경되는 테이블이나 환경변수가 0건이다.
- **SC-007**: 이미 설정된 프로젝트에 적용을 재실행하면 100% 변경이 없다(멱등).

## Assumptions

- 이 스킬은 axhub 의 앱 소유 동적 테이블과 앱 환경변수를 대상으로 한다. 외부 연결 데이터 소스에 대한 읽기는 인식하되 새 테이블로 추천하지 않는다.
- 다른 axhub 스킬과 일관되게, 이 스킬은 개발 라이프사이클 순간의 자동 제안과 명시적 자연어 요청 양쪽으로 활성화된다.
- 채택된 범위는 "추천 + 승인 시 적용"이다. 순수 분석(추천)은 인증 없이 항상 가능하고, 적용에는 인증과 대상 앱이 필요하다.
- 추론은 선언적 아티팩트(schema.prisma, 마이그레이션 파일, .env.example 등)를 1급·고신뢰로, 코드로만 정의된 모델(흩어진 ORM 클래스·런타임 getenv)은 베스트에포트로 처리한다. axhub 템플릿 스택(Next.js/TypeScript + FastAPI/Python)을 우선 지원하고, 특이하거나 완전히 동적인 패턴은 놓칠 수 있어 커버리지 갭으로 보고된다.
- 컬럼·타입·제약 추론은 베스트에포트다. 저신뢰 추측은 조용히 적용하지 않고 사용자 검토용으로 플래그한다.
- 추천 결과는 휘발성으로 다루며 파일로 저장하지 않는다. 영구화는 적용된 변경에만 일어나고, 다시 추천이 필요하면 소스를 재분석한다.
- 변경 적용에는 기존 테이블·환경변수 관리 기능을 재사용한다. 이 기능은 새로운 mutation 경로를 만들지 않는다.
